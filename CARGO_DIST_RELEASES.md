# Cargo Dist Release Guide

This project uses `cargo-dist` to build and publish precompiled binaries to GitHub Releases.

The goal of this setup is:

- users install with a one-liner
- GitHub Actions builds release binaries automatically
- you do not build and upload binaries by hand

## What Is Configured

These files control the release process:

- [Cargo.toml](/home/prompt-engineer/RustProjects/nexus-tui/Cargo.toml)
- [dist-workspace.toml](/home/prompt-engineer/RustProjects/nexus-tui/dist-workspace.toml)
- [.github/workflows/release.yml](/home/prompt-engineer/RustProjects/nexus-tui/.github/workflows/release.yml)
- [install.sh](/home/prompt-engineer/RustProjects/nexus-tui/install.sh)
- [install.ps1](/home/prompt-engineer/RustProjects/nexus-tui/install.ps1)

## How The Flow Works

1. You update the crate version in `Cargo.toml`.
2. You push the code to GitHub.
3. You create and push a Git tag like `v0.1.0`.
4. GitHub Actions runs the `Release` workflow.
5. `cargo-dist` builds archives and installer scripts.
6. The workflow uploads everything to a GitHub Release.
7. Your bootstrap scripts on `main` download the latest release installer.

## Important Files

### `Cargo.toml`

This must contain:

- the package version
- the `repository` URL
- the `license`
- the `readme`
- the `[profile.dist]` section

The release tag must match the version here.

Example:

- `version = "0.1.0"`
- tag: `v0.1.0`

### `dist-workspace.toml`

This file tells `cargo-dist`:

- which CI backend to use
- which installer types to generate
- which target platforms to build
- where installers should place binaries

Current targets:

- `aarch64-apple-darwin`
- `x86_64-apple-darwin`
- `x86_64-unknown-linux-gnu`
- `x86_64-pc-windows-msvc`

Current installers:

- `shell`
- `powershell`

### `release.yml`

This workflow is generated by `cargo-dist`.

It does not publish on every push to `main`.
It publishes when you push a version tag.

The workflow currently runs on:

- `pull_request`
- `push` to matching version tags

## What Gets Published

For each release, `cargo-dist` publishes:

- `ani-nexus-tui-installer.sh`
- `ani-nexus-tui-installer.ps1`
- `ani-nexus-tui-x86_64-unknown-linux-gnu.tar.xz`
- `ani-nexus-tui-x86_64-pc-windows-msvc.zip`
- `ani-nexus-tui-x86_64-apple-darwin.tar.xz`
- `ani-nexus-tui-aarch64-apple-darwin.tar.xz`
- checksum files
- source archive

Your top-level bootstrap scripts:

- `install.sh`
- `install.ps1`

do not build the app themselves. They download the generated release installer from the latest GitHub Release.

## Normal Release Process

Use this process every time you want to publish a new version.

### 1. Update The Version

Edit [Cargo.toml](/home/prompt-engineer/RustProjects/nexus-tui/Cargo.toml) and bump:

```toml
version = "0.1.1"
```

### 2. Commit And Push The Version Change

Example:

```bash
git checkout main
git pull
git add Cargo.toml
git commit -m "Bump version to 0.1.1"
git push origin main
```

If the version bump is part of a feature branch, merge that branch first.

### 3. Create The Release Tag

The tag should be the Cargo version with a leading `v`.

Example:

```bash
git tag v0.1.1
git push origin v0.1.1
```

### 4. Wait For GitHub Actions

Go to the repo Actions tab and open the `Release` workflow run created by the tag push.

You should see these jobs:

- `plan`
- `build-local-artifacts`
- `build-global-artifacts`
- `host`

The release is not complete until the workflow finishes successfully.

### 5. Verify The GitHub Release

Open the Releases page and confirm the tag has assets attached.

At minimum, confirm these are present:

- `ani-nexus-tui-installer.sh`
- `ani-nexus-tui-installer.ps1`
- Linux archive
- Windows archive
- macOS archives

### 6. Test The Install Commands

After the release is live, test:

```bash
curl -fsSL https://raw.githubusercontent.com/OsamuDazai666/nexus-tui/main/install.sh | bash
```

```powershell
irm https://raw.githubusercontent.com/OsamuDazai666/nexus-tui/main/install.ps1 | iex
```

These should now install from the latest GitHub Release, not compile from source.

## How To Cut A Release From A Feature Branch

Recommended flow:

1. Open PR from your feature branch.
2. Merge into `main`.
3. Pull latest `main`.
4. Verify `Cargo.toml` version is correct.
5. Create and push the tag from `main`.

Example:

```bash
git checkout main
git pull origin main
git tag v0.1.1
git push origin v0.1.1
```

Do not tag an old commit by mistake. Tag the exact commit you want released.

## How To Check If A Release Finished

The release is successful if both are true:

- the `Release` workflow run is green
- the GitHub Release page shows the generated assets

If the tag exists but no binaries appear, the workflow likely failed.

## How To Update Cargo Dist Itself

If you want a newer `cargo-dist` version:

1. install or update `dist` locally
2. rerun `dist init` or `dist generate`
3. review changes to `dist-workspace.toml` and `.github/workflows/release.yml`
4. commit those changes

Example:

```bash
curl --proto '=https' --tlsv1.2 -LsSf https://github.com/axodotdev/cargo-dist/releases/latest/download/cargo-dist-installer.sh | sh
dist init -y
dist generate
```

After that:

```bash
git add dist-workspace.toml .github/workflows/release.yml Cargo.toml
git commit -m "Update cargo-dist configuration"
git push
```

## Useful Local Commands

### Preview the release plan

```bash
dist plan
```

This shows which archives and installer files will be produced.

### Regenerate the GitHub workflow

```bash
dist generate
```

### Reinitialize config with defaults

```bash
dist init -y
```

Be careful with this. Review the generated changes before committing.

## Common Problems

### Tag does not trigger a release

Check:

- the tag was pushed to GitHub
- the tag matches the Cargo version
- the tag format is valid, for example `v0.1.1`

### Workflow runs but release is not published

Check:

- GitHub Actions is enabled for the repo
- the workflow has permission to write releases
- `GITHUB_TOKEN` has `Contents: Read and write`

### Assets are missing

Check the failed job logs in the `Release` workflow:

- `plan`
- `build-local-artifacts`
- `build-global-artifacts`
- `host`

### Install script fails

Check:

- the GitHub Release exists
- the installer asset names are still:
  - `ani-nexus-tui-installer.sh`
  - `ani-nexus-tui-installer.ps1`
- `install.sh` and `install.ps1` still point to the correct repo

If you rename the package or binary later, you may need to update:

- `Cargo.toml`
- `dist-workspace.toml`
- `install.sh`
- `install.ps1`
- README install examples

## Versioning Rules

Keep these in sync:

- `Cargo.toml` package version
- git tag
- GitHub Release version

Examples:

- `Cargo.toml`: `0.1.1` -> tag `v0.1.1`
- `Cargo.toml`: `1.2.0` -> tag `v1.2.0`

## Recommended Release Checklist

Before release:

- code merged to `main`
- `Cargo.toml` version updated
- tests pass
- install scripts still point at the correct repo

Release:

- create tag
- push tag
- wait for `Release` workflow

After release:

- verify assets exist on GitHub Releases
- test Linux/macOS install one-liner
- test Windows PowerShell install one-liner
